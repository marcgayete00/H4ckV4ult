    Reverse Engineering and debugging

    Behaviour detection

      In order to detect what you PE is doing on backguards it is highly recomended to use procmon filtering by the process name there you'll be able to check if the process creates and deletes files and more

      Disable file removal in Windows Temp

        In some cases the executable will create files in C:\Users\$username\AppData\Local\Temp\ this folder deletes some files by default so in order to disable that function we can check the following

        • Properties -> Security -> Advanced -> cybervaca -> Disable inheritance -> Convert inherited permissions into explicit permissions on this object -> Edit -> Show advanced permissions
        • Deselect the Delete subfolders and files, and Delete checkboxes.

    Debugging a PE (.exe)

      In this section we are documenting some informatation that might be useful when debugging with x64dbg

      Setup

        We will use x64dbg. When debugging with x64dbg it's recommended to select the Exit Breakpoint (Reboot required to apply changes). To do so go to Options -> Preferences and uncheck everything except Exit Breakpoint:

      Credential gathering

        Once your PE have been loaded click on the tab Memory Map here you should look for a register which is Type MAP and has a protection set to -RW--.

        Once you find it double click on it and look for the magic byte in the lower part. In case there is MZ means that file is a DOS MZ executable

        In order to extract that right click over the address and select Dump Memory to File

        Then use strings over the file you have exported in order to retrieve some data and credentials might be there

        C:\TOOLS\Strings\strings64.exe .\restart-service_00000000001E0000.bin

    Debugging a .jar

      jar files are archives so we can click over the file and unarchive all the files that contain. In case we would like to search for a specific string on the folder where are all the files

      ls fatty-client\ -recurse | Select-String "8000" | Select Path, LineNumber | Format-List

      One of the files that may contain interesting information is the beans.xml

      Rebuilding .jar file Basic

        In case we would like to perform any modifications in the main files (.xml) not in the code itself, we can do it but we need to have in mind that the beans.xml may contain some kind of salting which will be appended in all file hashes which then will be checked with the base file hash to check if there was any modifications. Here is an example of that

        <bean id="secretHolder" class = "htb.fatty.shared.connection.SecretHolder">
      <property name = "secret" value = "clarabibiclarabibiclarabibi"/>
</bean>

        To bypass this checks we need to remove all hashes present in the file META-INF/MANIFEST.MF and delete 1.RSA and 1.SF This may vary depending of the jar file. Here is an example of how the MANIFEST.MF should be. Remember to leave a blank space at the end

        Manifest-Version: 1.0
Archiver-Version: Plexus Archiver
Built-By: root
Sealed: True
Created-By: Apache Maven 3.3.9
Build-Jdk: 1.8.0_232
Main-Class: htb.fatty.client.run.Starter

        Once done compile the java again

        jar -cmf .\META-INF\MANIFEST.MF ..\fatty-client-new.jar *

      Rebulding .jar file Advanced

        If we would like to perform any mofification in the java code we need to use a java debugger. In this case we will use JD-GUI. Just open it and select your .jar file

        From inside you'll be able to view the code and also to export it to make some modifications. To do press the button Save All Sources

        At this point you'll have access to all the code file so open any of then and perform all the modifications

        Now use the following command to compile the file you modified using an external library which in this case is the original jar file we have downloaded previously. This will create some class files inside the directory of the file you are compiling

        javac -cp fatty-client-new.jar Newjar\htb\fatty\client\gui\ClientGuiTest.java

        Now create a new folder (folder name: raw) and copy and paste the original .jar file inside this folder. Then extract it's contents by right clicking on it and selecting extract here

        Now we will copy all the classes that we generated previously to the folder where our new .jar is being builded (folder name: raw) 

        mv -Force newjar\htb\fatty\client\gui\*.class raw\htb\fatty\client\gui\

        Finally access the new folder (raw) and compily a new jar using all the content you copyed. This way a new jar file will be created. Double click to open it and test your new modifications

        jar -cmf META-INF\MANIFEST.MF traverse.jar .

    Debugging a ELF

      In order to check if a file is an ELF we can can use the following command

      file octopus_checker 
octopus_checker: ELF 64-bit LSB shared object, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2,...

      To debug this file we can use dbg. Here are some commands to use it. First open the file

      gdb ./octopus_checker

      Then configure the disassembly-flavour to define the display style of the code and proceed with disassembling the main function of the program. 

      disas main 
set disassembly-flavor intel

      In this case we saw a SQL Driver connection string so we configured a breakpoint there

      0x0000000000001607 <+433>:   call   0x11b0 <SQLDriverConnect@plt>
   0x000000000000160c <+438>:   add    rsp,0x10
   0x0000000000001610 <+442>:   mov    WORD PTR [rbp-0x4b4],ax
   0x0000000000001617 <+449>:   lea    rsi,[rip+0xa70]        # 0x208e
   0x000000000000161e <+456>:   lea    rdi,[rip+0x2a1b]        # 0x4040 <_ZSt4cout@@GLIBCXX_3.4>

      To setup the breakpoint use directly the string and the software itself will calculate the memory address. But you can specify the memory address itself 

      break SQLDriverConnect@plt   #Using the string 
b* 0x0000000000001607   #Using the memory adress

      In case you would like to view all breakpoints and remove any of them we can do.

      info breakpoints 
delete 1

      Finally just run the code with the breakpoint. In case it fails try to change the way you are setting up the breakpoint.

      This will automatically show the code inside PLEASE READ CAREFULLY SINCE THERE IS A LOT OF CONTENT AND YOU MAY MISS SOMETHING

      Debugging a .dll

        In case you found a DLL file and you would like to check it's content you should use dnspy since this files are written in .NET you'll be able to read the code with any issues unless is offuscated. But in that case use the de4dot.exe

    Desofuscation

      Many times will find out that the code we have is offuscated. There are several softwares which we can use to solve that

      .NET PE (Executable)

        In case we find out that the file we have looks like a .NET executable (Run strings over the file and look for NETFramework) we can use De4dot to solve that. The following output shows the command we can use to desofuscate a file

        de4dot.exe C:\Users\auditor\Desktop\restart-service_00000000026A0000.bin

        Once done we can use dnspy to view the content of the .NET executable just by opening the file and going to Executable_name.exe -> {} Executable_name.exe
